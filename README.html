Identifying Relationship-level Effects Using Covariance Restrictions

De Jonghe, O. and Lewis, D. (2026). Identifying Relationship-level Effects Using Covariance Restrictions. ECB Working Paper. [Number forthcoming.]

This page provides the code and a sample dataset for the estimation procedure in De Jonghe and Lewis (2026). The paper proposes a method to identify relationship-level supply and demand shocks in many-to-many bipartite networks, using covariance restrictions rather than the homogeneity assumptions of the standard AKM framework (Abowd, Kramarz & Margolis, 1999). The method is general and applicable to any bipartite network setting — worker–firm matches, buyer–supplier relationships, and beyond. The code provided here implements the application to credit markets: given observed changes in loan quantities and interest rates for firm–bank pairs, the estimator recovers a relationship-specific bank loan supply shock and a relationship-specific firm loan demand shock for each pair, without requiring external instruments — generalising approaches such as Khwaja & Mian (2008) and Amiti & Weinstein (2018) to allow for heterogeneous, relationship-level effects. Identification comes from exploiting cross-firm and cross-bank covariance matrices of the joint price–quantity changes.

Files

File Type Description
python_Supply_Demand_AKM_restrict_LamFFnorm.py Python script Core estimation routine. Reads a long-format panel of firm–bank price and quantity changes, estimates the 2×2 mixing matrix A, decomposes observed changes into supply and demand shocks, and runs the AKM overidentification test. Writes two output files: the input data augmented with shock columns, and a one-row summary of results.
csv_FBT_panel_QP_long.csv CSV data Sample input dataset in long format. Each row is a firm–bank–time observation with columns: Firm, Bank, Time, dQdh (quantity change), dP (price change). Provided to allow the Python script to be tested immediately, without access to the underlying confidential data.
001_stata_python_integrated_estimation.do Stata do-file Example of how the Python script is called inside a full empirical workflow. Loops over countries and time periods: loads and merges raw data, winsorizes and partials out fixed effects and relationship controls via reghdfe, exports the minimalistic CSV for Python, calls the Python script via a shell command (passing country and period as environment variables), reads back the shock output, and runs sanity checks before saving. Intended as a template; users working in R or another language can adapt the logic straightforwardly.

Quick Start — Python Script Standalone

To verify that the script runs and to inspect its output, you only need the Python script and the sample CSV. Both files must be in the same folder.

Requirements

  • Python 3.8 or later
  • Packages: pandas, numpy, scipy, joblib

Install missing packages with:

pip install pandas numpy scipy joblib

Run

python python_Supply_Demand_AKM_restrict_LamFFnorm.py

The script will print progress to the console and produce two output files in the same folder:

  • csv_FBT_panel_QPSD_long.csv — the input data with appended Supply and Demand shock columns
  • summary_output.csv — a single-row summary of the estimation results for this run

User Configuration

At the top of the script there is a clearly marked user configuration block. The defaults work out of the box with the sample CSV. The configurable parameters are:

ParameterDefaultWhat it does
country, start_period, end_period read from environment variables Purely for labelling: these identifiers are written into summary_output.csv so that results from multiple runs can be traced back to their sample. They do not affect the estimation. When running standalone, they can be set to any string directly in the script.
input_file csv_FBT_panel_QP_long.csv Path to the input CSV. Can be a filename (if the CSV is in the same folder) or an absolute path.
output_shocks_file, output_summary_file csv_FBT_panel_QPSD_long.csv, summary_output.csv Names of the two output files. The shocks file is overwritten on each run; the summary file is appended.
n_jobs -1 Number of CPU cores for the parallelised bank loop. -1 uses all available cores. Set to 1 to disable parallelisation (useful for debugging or on shared servers).
random_seed 42 Random seed for the AKM overidentification test. Fix for reproducibility; set to None to disable.
nstarts 100 Number of random starting values used in the AKM test optimisation. Higher values give a more reliable minimum at the cost of computation time; 100 is generally sufficient.
significance_level 0.10 Significance level for the chi-squared rejection threshold in the AKM test (e.g. 0.10 = 10%).

Integrating into a Stata Workflow

The do-file illustrates the full pipeline used in the paper. The key steps for each country–period cell are:

  1. Load and merge the raw firm–bank panel with sector identifiers, firm–bank relationship characteristics, and borrower-sector-region-time controls.
  2. Winsorize dQdh and dP within each period.
  3. Partial out time fixed effects and lagged relationship characteristics using reghdfe; save residuals (scaled ×100) as the new dQdh and dP.
  4. Create contiguous integer indices for Firm, Bank, and Time using gegen group().
  5. Export the five-column CSV (Firm, Bank, Time, dQdh, dP).
  6. Call Python via a shell command, passing country and period as environment variables:
shell cmd /c "set COUNTRY=`country' && set START_PERIOD=`start_period' && set END_PERIOD=`end_period' && python "path/to/python_Supply_Demand_AKM_restrict_LamFFnorm.py""
  1. Read back csv_FBT_panel_QPSD_long.csv and merge the supply and demand shocks into the full preserved dataset.
  2. Sanity check: verify that the correlation between the Stata and Python copies of Q and P exceeds 0.99999 and that the merge is 100% matched before saving.

Users working in R or another environment can follow the same logic: the only interface between the statistical environment and the Python script is the five-column CSV in and the seven-column CSV out.

Input Format

ColumnDescription
FirmInteger firm identifier (contiguous, starting at 1)
BankInteger bank identifier (contiguous, starting at 1)
TimeInteger time identifier (contiguous, starting at 1)
dQdhQuantity change (e.g. growth in loan volume), after any desired cleaning and partialling out
dPPrice change (e.g. change in interest rate), after any desired cleaning and partialling out

Not every firm–bank–time combination needs to be present. The script handles unbalanced panels. The identifiers must be contiguous integers starting at 1 (or 0); the do-file uses gegen group() to create these from arbitrary raw identifiers.
